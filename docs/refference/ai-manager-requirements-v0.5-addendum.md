# AI マネージャー要件定義 v0.5 追補 — 管理者発火の状況確認

> **位置づけ:** v0.2〜v0.4 への追加要件。矛盾時は本書を優先する。
> **背景:** オペレーター要望(2026-07-10)「管理者の意志(ダッシュボードのボタン押下)で発火する、現在の進捗や状況の問いかけ機能がほしい」。

## 1. 要件

ダッシュボードの管理者限定ページに **状況確認**(`/admin/checkin`)を追加する。

| 機能 | 内容 |
|---|---|
| メンバー一覧 | active な member の一覧+当日の対話状況(朝の問いかけへの応答有無等)を表示 |
| 個別に問いかけ | ボタン押下で対象メンバーの Chat DM に「現在の進捗・状況の問いかけ」を送信 |
| 全員に問いかけ | active な member 全員へ一括送信 |

> **改訂(v0.8 §3):** 配信対象・一覧の範囲は [v0.8 追補](./ai-manager-requirements-v0.8-addendum.md)で
> 「role = 'member' 固定」から「ユーザー単位の問いかけ可否フラグ(checkin_enabled)」に変更された。
> 上表は v0.5 時点の記録である。

- 問いかけ文面は本人のタスク状況を踏まえて生成(LLM 不調時は定型文へフォールバック — 朝の問いかけと同じ設計)
- 冒頭に管理者発火であることを明示する(例: 「(管理者からの状況確認です)」)
- メンバーの返信は通常の対話として継続し、全件 ops.dialogues に構造化保存される(M2/M4/M5 の入力になる)

## 2. 設計原則(確定判断)

1. **配信の実行主体は batch**(朝の問いかけと同一経路)。ダッシュボードは「今すぐ同期」と同じ
   OIDC 呼び出しパターンで batch を起動するだけとし、dashboard に Chat 送信権限や
   ops.dialogues への書込権限を持たせない(権限境界を増やさない)
2. 朝の問いかけの「1日1回」冪等ガードは適用しない(管理者の意志による都度発火が目的)。
   ただし二重送信はボタン側(PRG)で防止する
3. DM スペース未登録(chat_space_id なし)のメンバーはスキップし、結果を画面に表示する
4. 全操作を監査ログに記録する(誰が・誰に・いつ)
5. **仮説形成の対話を横取りしない**: 朝の問いかけ・完了時の振り返りに応答中
   (open な対話に本人の返信が付いている状態)のメンバーはスキップし、結果を画面に表示する。
   返信の帰属は「当日の最新の open な対話」に決まるため、途中で状況確認を差し込むと
   本人の続きの返信が状況確認側に吸収され、仮説形成(M2)のフローが中断するため。
   まだ返信が付いていない「未応答」のメンバーには通常どおり送信できる
   (未応答者への問いかけが本機能の主目的)。
   なお完了時の振り返りは本人の完了申告(本人の発話)から始まる対話のため、
   open である間は常にスキップ対象になる。応答の途中で離脱した open な対話が残ると
   その日のうちは当該メンバーへ送信できないが、仮説形成の保護を送信の利便性より
   優先するトレードオフとして許容する

## 3. スコープ外

- メンバーからの返信の督促・リマインド(送りっぱなしとし、返信状況は既存ダッシュボードで確認)
- 問いかけ文面の画面からのカスタマイズ(将来検討)

## 4. 受け入れ基準

1. 管理者がボタンを押すと対象メンバーの DM に状況確認が届き、返信が対話として保存される
   (§2-3 の DM 未登録・§2-5 の応答中によるスキップ対象を除く)
2. DM 未登録メンバーと、朝の問いかけ・振り返りに応答中のメンバーはスキップされ、
   その旨が画面に表示される
3. member ロールの利用者は本ページにアクセスできない

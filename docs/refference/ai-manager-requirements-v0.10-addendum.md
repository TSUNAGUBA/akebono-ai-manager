# AI マネージャー要件定義 v0.10 追補 — プロジェクト計画情報・タスク進捗管理・AI への文脈供給

> **位置づけ:** 本書は `ai-manager-requirements-design.md`(v0.2)への**追補**であり、v0.2 本文は変更しない。
> 既存の追補(v0.3〜v0.9)と本書が矛盾する場合は本書(v0.10)を優先する。
> **背景:** オペレーター要望(2026-07-13)「プロジェクトの内容や目的、マイルストーン、各タスクと
> その進捗の管理もできるようにしたい。すべてのプロジェクトが詳細に計画されているとは限らないため、
> 任意項目を残しつつ、あれば管理できる機能が望ましい。これらの情報は AI に渡されたうえで対話が
> なされるようにする。プロジェクトと顧客・プロジェクトとタスクの関係が混ざらないよう注意する」。

## 1. 背景と課題

| # | 課題 | 現状 |
|---|---|---|
| C11 | プロジェクトに計画情報を持てない | ops.projects は名称・顧客・種別・状態・優先度・担当のみ(v0.9)。内容・目的・マイルストーンを記録する場所がない |
| C12 | タスクの進捗をダッシュボードから管理できない | 進捗更新は Chat の対話(M3)のみ。プロジェクト単位でタスクを見渡して状態を直す手段がない |
| C13 | AI がプロジェクトの文脈を知らない | 対話(朝の問いかけ・状況確認・QA)にはタスク状況しか供給されず、プロジェクトの目的やマイルストーンを踏まえた壁打ちができない |

## 2. プロジェクト計画情報(C11 対応)

- `ops.projects` に **内容(description)・目的(objective)** を追加する。**すべて任意項目**
  (詳細に計画されていないプロジェクトを許容する — NOT NULL を付けない、UI でも必須にしない)
- **マイルストーン**(`ops.project_milestones`)を追加する: 名称(必須)・期日(任意)・
  状態(planned / done)。**プロジェクトにのみ属する**(§5)
- 管理はプロジェクト編集ページ(/admin/projects?edit=)に統合する: 計画情報の入力、
  マイルストーンの追加・完了切替・削除
- AI への供給(§4)では、**未入力の任意項目は出力から省略する**(「未登録」の羅列で
  プロンプトを汚さない)

## 3. タスク進捗の管理(C12 対応)

- プロジェクト編集ページに、そのプロジェクトに属するタスクの一覧(題名・担当・期限・状態)と
  **状態の更新**(proposed / approved / in_progress / blocked / done / cancelled)を追加する
- 状態遷移は `ops.task_status_log`(changed_via='admin')へ**同一トランザクション**で記録する
  (既存のタスク状態遷移の原則を踏襲。同一状態への更新は no-op で履歴を汚さない)
- **タスクの起票・題名・担当・期限の変更は行わない**: タスク作成は M3(Chat のタスク指示 →
  AI 分解 → 管理者承認)が SoT のまま。UI はプロジェクト単位の可視化と進捗修正に限定する
- 権限: `ai_manager_admin_rw` へ ops.tasks の SELECT+**status 列(および updated_at /
  completed_at)のみの UPDATE**+task_status_log の SELECT / INSERT を追加(ADR-16)

## 4. AI への文脈供給(C13 対応)

プロジェクトの計画情報を、SoT(ops)からプロンプト組み立て時に直接取得して供給する
(ADR-13 と同じ判断: rag への文書化同期はしない。取得失敗は非ブロッキング — 原則4)。

### 4.1 朝の問いかけ・状況確認(配信・対話継続)

本人の**未完了タスク(approved / in_progress / blocked)が属する進行中(active)プロジェクト**の
計画情報(名称・顧客名・目的・内容・マイルストーン)を「プロジェクト文脈」ブロックとして供給する。
プロンプトは、目的・マイルストーンを「位置づけ」の壁打ちに活用するよう指示する。

- 供給対象は **status = 'active' のプロジェクトのみ・優先度順に最大 5 件**
  (終了プロジェクトはタスク指示の分解候補と同様に文脈から外す。上限はタスク文脈の
  LIMIT と同様、プロンプトの際限ない肥大の防止)。QA(§4.2)での明示的な言及は
  終了プロジェクトでも回答対象とする(質問への回答を妨げない)

### 4.2 随時 QA

対象プロジェクトの特定は対象顧客の特定(v0.7 §4)と同じ優先順で行う:
①質問文中のプロジェクト名/ID のマスタ照合(最長一致・LIKE エスケープ・2文字以上)
②対話文脈のプロジェクト(本人の直近 in_progress タスクのプロジェクト)。
特定できたら「プロジェクト情報」ブロックとして計画情報を供給する。

## 5. データ境界(混同の防止)

| 関係 | 定義 | プロンプト上の扱い |
|---|---|---|
| プロジェクト ↔ 顧客 | projects.customer_id(任意 FK)。プロジェクトは高々1顧客に属する | プロジェクトブロックには**顧客名のみ**を属性表示。顧客の取引先・業界等は「顧客マスタ情報」(v0.7)の別ブロックが担い、相互に混ぜない |
| プロジェクト ↔ タスク | tasks.project_id(任意 FK)。タスクは高々1プロジェクトに属する | プロジェクトブロックにタスク一覧は含めない(「本人のタスク状況」ブロックが担う) |
| プロジェクト ↔ マイルストーン | project_milestones.project_id(必須 FK)。マイルストーンはプロジェクトにのみ属する(タスク・顧客に紐づけない) | プロジェクトブロック内にのみ表示 |
| QA での特定 | 顧客の特定とプロジェクトの特定は**独立に**行う | 「A社SI の目的は?」→ プロジェクト情報、「A社の取引先は?」→ 顧客マスタ情報。両方に言及があれば両方供給(混同せず、欠落もさせない) |

- UI のマイルストーン・タスク操作は**すべて project_id を WHERE 条件に含め**、
  別プロジェクトの対象を操作できないようにする(hidden input 偽装への防御)

## 6. スコープ外(本追補では実施しない)

- タスクの起票・題名・担当・期限の UI 変更(M3 の Chat 承認フローが SoT)
- マイルストーンとタスクの紐づけ(マイルストーン配下のタスク管理) — 必要性が確認された時点で判断
- メンバー向け(非管理者)のプロジェクト計画閲覧 UI(既存の閲覧ページの拡張は将来検討)
- 週次サマリ・日報へのマイルストーン反映(レポート系は現行のまま。運用で必要になれば追加)

## 7. 受け入れ基準

1. プロジェクトの内容・目的・マイルストーンを未入力のままでも登録・運用できる(任意項目)
2. 計画情報を入力すると、朝の問いかけ・状況確認・QA の AI 文脈に供給される
   (未入力の項目はプロンプトに現れない)
3. プロジェクト編集ページから所属タスクの状態を更新でき、遷移が task_status_log に
   changed_via='admin' で記録される
4. 別プロジェクトのマイルストーン・タスクを操作できない(project_id 不一致は拒否)
5. QA でプロジェクト名を含む質問にはプロジェクト情報が、顧客名を含む質問には顧客マスタ情報が、
   それぞれ独立に供給される
6. 下位互換: 既存プロジェクト・タスクへの影響なし(列追加+新テーブルのみ。データ更新パッチ不要)。
   UI からタスクの起票・削除・題名変更はできない(権限でも担保)

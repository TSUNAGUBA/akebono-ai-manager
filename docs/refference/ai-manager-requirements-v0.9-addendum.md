# AI マネージャー要件定義 v0.9 追補 — プロジェクト管理 UI・優先順改訂・エイリアス照合・対話の耐障害性

> **位置づけ:** 本書は `ai-manager-requirements-design.md`(v0.2)への**追補**であり、v0.2 本文は変更しない。
> 既存の追補(v0.3〜v0.8)と本書が矛盾する場合は本書(v0.9)を優先する
> (§3 は v0.8 §3.4、§4 は v0.3 §4.3 の別名対応、の各改訂を含む)。
> **背景:** オペレーター指示と運用インシデント 2 件(いずれも 2026-07-13)に基づく要件化。

## 1. 背景と課題

| # | 課題 | 現状 | 影響 |
|---|---|---|---|
| C7 | プロジェクト登録の UI がない | `ops.projects` への登録は SQL 直接実行のみ(v0.3 マスタ管理 UI の対象外) | 非エンジニアがプロジェクトを登録できない。運用ミスの温床(v0.3 C3 と同根) |
| C8 | 明示的なタスク指示が進行中対話に吸収される | 進行中対話の継続がタスク指示検知より常に優先(M3) | 問いかけ可の管理者が朝の問いかけに未返信のまま「タスク: …」を送ると起票されない(v0.8 §3.4 の既知の制約) |
| C9 | 法人格付きの顧客名が質問文と照合できない | 照合は「質問文 ⊇ 登録名」方向の部分一致のみ | 名称「株式会社しまむら」の顧客を「しまむらの取引先は?」から特定できず、マスタ登録済みでも「見当たらない」と回答(2026-07-13 再発インシデント)。v0.7 の顧客マスタ情報供給が機能しない |
| C10 | 問いかけへの返信処理が脆弱 | 対話継続(朝・夕・状況確認)の応答生成失敗は例外がそのまま伝播 | ユーザーに汎用エラーが返り、**返信テキストがどこにも記録されず失われる**(2026-07-13 返信エラーインシデント)。配信側には定型文フォールバックがあるのに継続側にはない非対称 |

## 2. プロジェクト管理 UI(C7 対応)

ダッシュボードのマスタ管理に**プロジェクト**(`/admin/projects`)を追加する。

| 操作 | 内容 |
|---|---|
| 一覧 | 名称・顧客・種別・状態・優先度・担当管理者・更新日時 |
| 追加 | プロジェクトID(変更不可)・名称・顧客(任意)・種別・状態・優先度(任意)・担当管理者(任意・admin ロールから選択) |
| 編集 | ID 以外の全属性 |

- **物理削除はしない**(タスク・対話ログ・dwh が参照)。終了したプロジェクトは状態を
  `closed`(終了)にする。`active` のみがタスク指示(M3)の分解候補になる(既存挙動を踏襲)
- 種別は DDL の CHECK 制約(si / saas / media / utsuwa / internal)と同じ値集合を UI で提示する
  (種別のマスタ化は現状の規模では過剰なため見送り。追加が必要になった時点で v0.3 業界マスタと
  同じパターンでマスタ化を検討する)
- **既存データの保護(原則7):** status には DDL の CHECK がなく SQL 直登録の値があり得るため、
  編集フォームはリスト外の現在値を「(現在値)」として選択肢に残し、サーバー側も現在値の温存を
  許可する(黙った巻き戻り防止)。担当管理者が無効化・降格済みの場合も現担当を選択肢に残す
- 担当管理者はサーバー側でも admin ロールであることを検証する(FK は ops.users 全体を許すため)
- 権限: `ai_manager_admin_rw` に ops.projects の SELECT / INSERT / UPDATE を追加(DELETE は付与しない)。
  担当管理者の選択肢は既存の ops.users 列単位 SELECT(v0.8 §4)の範囲で取得する
- セキュリティは既存のマスタ管理と同一(管理者限定・CSRF・監査ログ・PRG)。モバイル対応は原則8に従う

## 3. タスク指示と対話継続の優先順の改訂(C8 対応・v0.8 §3.4 の解消)

MESSAGE ハンドラの優先順を以下に**改訂**する:

1. 裁定の受領(管理者・M6)
2. 提案理由の受領
3. **明示的なタスク指示(管理者・ルールベース確定シグナルのみ)** ← 新設(進行中対話より優先)
4. 進行中の朝夕対話・状況確認の継続
5. 曖昧なタスク指示の分類(管理者・flash-lite)
6. 完了申告の検知
7. 随時 QA

> **判断理由:** 「タスク:」等の明示プレフィックスは誤検知の余地がない確定シグナルであり、
> 対話保護(横取り防止)の必要がない。一方、担当者・期限・依頼動詞のヒューリスティックによる
> **曖昧な**シグナルは従来どおり対話の継続を優先する(誤起票より対話保護)。
> これにより v0.8 §3.4 の既知の制約(管理者が朝の問いかけに未返信だと指示が吸収される)は
> 明示プレフィックスを使う限り解消される。曖昧な表現の指示は引き続き対話クローズ後に送る運用。

## 4. 顧客エイリアス照合とエスカレーション診断情報(C9 対応)

### 4.1 エイリアス(設計 SoT: phase3-and-migration-design.md §2 — 本追補で実装)

- `ops.customer_aliases`(customer_id, alias。PK は複合、alias は 2 文字以上)を追加する
- 対象顧客の特定(identifyTargetCustomer)は**名称・顧客ID・エイリアスの UNION** に対して
  照合する(LIKE エスケープ・最長一致・2文字以上の既存ルールを踏襲)
- エイリアスの管理は顧客編集フォーム(/admin/customers)に統合する(読点・カンマ区切りの
  複数入力 → 洗い替え。customer_industries と同じトランザクションパターン)
- 権限: dashboard_ro に SELECT、admin_rw に SELECT / INSERT / DELETE を追加

### 4.2 エスカレーションの診断情報

確信度 low のエスカレーション本文に**対象顧客の特定結果**を含める
(特定済みなら customer_id、未特定なら「特定できず — 名称・エイリアスと不一致の可能性」)。
管理者が「マスタ未登録」「エイリアス不足」「関係未登録」のどれが原因かを切り分けられるようにする。

## 5. 対話継続・QA の耐障害性(C10 対応)

### 5.1 対話継続の回復(返信テキストの保護)

朝の問いかけ・夕の振り返り(開始含む)・状況確認への返信処理で応答生成(LLM)が失敗した場合:

1. **ユーザーの返信ターンを先に SoT(ops.dialogues)へ保存する**(記録系データの保護 — 原則2)。
   AI 側のフォールバックターンも併せて保存し、対話履歴の user/model 交互性を保つ
2. 汎用エラーではなく定型文で応答する(「返信は記録しています。少し時間をおいて続きを
   そのまま送ってください」)。対話は open のまま残り、次のメッセージで継続できる
3. 保存まで失敗した場合のみ例外を伝播させる(返信が記録できておらず再送を促す必要があるため)

補助文脈(本人のタスク状況・今朝の対話)の取得失敗も返信の処理を止めない(文脈なしで継続)。

> **既知の制約(許容):** 状況確認(adhoc_checkin)の最終往復で失敗した場合、フォールバックの
> 2 ターンでターン数上限に達して対話がクローズし、次のメッセージは通常のルーティング
> (QA 等)で処理される。返信は保存済みでデータ消失はないため許容する。

### 5.2 QA の補助処理の非ブロッキング化

随時 QA で以下の補助処理が失敗しても回答を止めない(原則4):

- ナレッジスコープの導出失敗 → 顧客固有を除外(安全側)して継続
- ナレッジ検索(embedding 依存)・例え話検索の失敗 → 参考情報なしで継続
  (顧客マスタ情報が供給されていれば構造の質問には回答できる)

### 5.3 エラー文言の診断性

やむを得ず汎用エラーを返す場合、想定エラー(AIM コード付き)であれば**エラーコードを文言に含める**。
管理者が逆引き表(docs/operations/error-codes.md)と Cloud Logging で即座に原因を特定できるようにする。

## 6. スコープ外(本追補では実施しない)

- プロジェクト種別・状態のマスタ化(§2 のとおり固定値集合を維持)
- タスクの一覧・編集 UI(従来どおり Chat の動線が SoT)
- エイリアスの自動生成(法人格の自動除去等) — 登録は管理者の明示操作とする
- 対話継続の LLM 失敗時の自動リトライ(定型文+再送誘導で足りる。頻発時に別途検討)

## 7. 受け入れ基準

1. プロジェクトの追加・編集(顧客・種別・状態・優先度・担当管理者)が SQL を書かずにダッシュボードから行える
2. 名称「株式会社しまむら」+エイリアス「しまむら」の顧客に対し、「しまむらの取引先は?」で
   マスタの関係に基づく回答が返る(v0.7 受け入れ基準1がエイリアス経由で成立する)
3. 問いかけ可の管理者が朝の問いかけに未返信のまま「タスク: …」を送るとタスク起票される。
   曖昧な指示らしいメッセージは従来どおり対話の継続として扱われる
4. 問いかけへの返信中に LLM が失敗しても、返信テキストが ops.dialogues に保存され、
   定型文の応答が返る(汎用エラー・返信消失にならない)
5. 確信度 low のエスカレーションに対象顧客の特定結果が含まれる
6. 下位互換: スキーマ追加(customer_aliases)のみで既存データへの影響なし。
   エイリアス未登録の顧客は従来どおり名称・顧客ID で照合される

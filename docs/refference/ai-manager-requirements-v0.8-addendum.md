# AI マネージャー要件定義 v0.8 追補 — 問いかけ対象のユーザー単位設定

> **位置づけ:** 本書は `ai-manager-requirements-design.md`(v0.2)への**追補**であり、v0.2 本文は変更しない。
> 既存の追補(v0.3〜v0.7)と本書が矛盾する場合は本書(v0.8)を優先する
> (§3 は v0.5 の配信対象、§4 は v0.3 §5.1 の権限境界の改訂を含む)。
> **背景:** オペレーター指示(2026-07-13)に基づく要件化。

## 1. 背景と課題

AI からの問いかけ(朝の問いかけ・管理者発火の状況確認)の配信対象は、これまでロールで固定されていた
(`role = 'member'` の active ユーザー全員。管理者は常に対象外)。

| # | 課題 | 現状 | 影響 |
|---|---|---|---|
| C6 | 問いかけ対象がロール固定 | morning-checkin / adhoc-checkin の対象クエリと状況確認画面が `role = 'member'` で固定 | メンバー個別の事情(長期離席・役割変更・本人の希望)で問いかけを止められない。管理者に問いかけを届けることもできない |

## 2. 設計判断

1. **ユーザー単位のフラグ(`ops.users.checkin_enabled`)で対象を選定する。** ロールは権限
   (管理機能へのアクセス)の軸であり、問いかけの要否とは独立した関心事である。
2. **既存ユーザーの初期値は従来動作を保存する**(member → 可 / admin → 不可)。挙動の変化は
   オペレーターが明示的に設定を変更したユーザーに限られる(下位互換・開発原則 7)。
   新規ユーザーの既定値は「可」(ロール不問。不要なら UI で無効化)。
3. **設定 UI はダッシュボードの管理者ページに置く**(手動 SQL を新たな運用手順にしない — 開発原則 1)。
4. **対象外のユーザーは配信ジョブの対象クエリで除外する**(自動配信・管理者発火の両方)。
   状況確認画面では不可のユーザーも一覧に表示し(可否バッジ付き)、送信ボタンのみ無効化する
   (「見えない」より「見えて操作できない+理由がわかる」を優先)。

## 3. 機能要件

### 3.1 配信対象の選定(v0.5 §2 の配信対象の改訂)

- 朝の問いかけ(morning-checkin)・状況確認(adhoc-checkin)の対象は
  **「active かつ問いかけ可(checkin_enabled)」のユーザー**とする。ロールでは絞らない
- 状況確認の個別送信時の対象検証(画面・batch の両方)も同条件とする
- DM 未登録・応答中スキップ等の既存のスキップ条件は変更しない

### 3.2 設定 UI(ダッシュボード /admin/users)

- 管理者限定。全ユーザーを名前・ロール・状態(active)・DM 登録・問いかけ可否付きで一覧表示する
- 操作は**問いかけ可否の切替のみ**(目標値を明示した更新のため二重送信でも冪等)。
  変更は監査ログ(誰が・誰を・どちらへ)に記録し、POST は CSRF トークン必須(v0.3 §5.1 踏襲)
- ユーザーの追加・名前・ロール等の編集は本追補のスコープ外(従来どおり SQL 運用を継続)
- モバイル表示は既存の管理ページと同じレスポンシブテーブルで対応(開発原則 8)

### 3.3 状況確認画面(/admin/checkin)の表示

- 一覧はロールで絞らず active な全ユーザーを表示し、問いかけ可否のバッジ列を追加する
- 問いかけ不可のユーザーは個別送信ボタンを無効化し、理由(設定変更の導線)を表示する

### 3.4 既知の制約: 管理者を問いかけ可にする場合の対話とタスク指示の干渉

> **改訂(v0.9 §3):** 本節の制約は [v0.9 追補](./ai-manager-requirements-v0.9-addendum.md)で
> 「明示的なタスク指示(『タスク:』等)は進行中対話より優先する」優先順の改訂により解消された
> (曖昧な指示らしいメッセージは従来どおり対話の継続を優先)。以下は v0.8 時点の記録である。

管理者を問いかけ可に設定すると、朝の問いかけが管理者にも届く。既存仕様(M3)では
**進行中対話の継続がタスク指示の検知より優先される**(朝夕対話の途中のメッセージが
タスク起票に横取りされないための設計)ため、管理者が朝の問いかけに**返信しないまま**
タスク指示(「タスク: …」等)を DM すると、その指示は朝の対話への返信として扱われ、
タスク起票(M3)が実行されない。

- **運用ガイダンス:** 問いかけ可の管理者は、朝の問いかけへ返信して対話を閉じてから
  タスク指示を出す(仮説が確定すると対話はクローズされる)。この干渉を避けたい管理者は
  問いかけ不可のままにする(既存管理者の初期値は不可)
- **本追補での判断:** 対話継続とタスク指示の優先順(M3 の設計判断)は変更しない。
  優先順自体の見直し(例: 明示プレフィックス付き指示は対話継続より優先する)は、
  運用で支障が確認された時点で別要件として検討する

## 4. 権限設計(v0.3 §5.1 の改訂)

書込ロール `ai_manager_admin_rw` に `ops.users` の**列単位の権限**を追加する:

- `SELECT (user_id, display_name, role, active, chat_space_id, checkin_enabled)` — 一覧表示用
  (email 等の他列には付与しない)
- `UPDATE (checkin_enabled)` — 可否切替用(他列の更新は不可。名前・ロール・active の変更は
  引き続き SQL 運用の権限でのみ可能)

> **判断理由:** v0.3 §5.1 の「マスタ 4 表+顧客のみ書込可」の境界は、ops.users への表レベル権限を
> 与えないまま維持する。列単位 GRANT により、UI 経由で変更できる範囲を機能要件(可否切替)と
> 正確に一致させる。GRANT は repeatable マイグレーション(30_grants.sql)が冪等に適用するため、
> 新たな手動手順は発生しない。

## 5. データモデル・移行(開発原則 7)

```sql
ALTER TABLE ops.users ADD COLUMN checkin_enabled BOOLEAN NOT NULL DEFAULT TRUE;
UPDATE ops.users SET checkin_enabled = (role = 'member');  -- 従来動作の保存
```

- versioned マイグレーション(0007)。初期値の設定は一度だけ実行され、適用後に UI で変更した
  設定が再実行で巻き戻ることはない(開発原則 2)
- seed テンプレート(seed-users.sample.sql)は checkin_enabled を**初期投入時のみ**設定し、
  ON CONFLICT の更新対象に含めない(再実行が UI の設定を巻き戻さない)
- ロールバックは列削除(`ALTER TABLE ops.users DROP COLUMN checkin_enabled;`)で可能。
  設定値は失われる(復元不能な点として明示)

## 6. スコープ外(本追補では実施しない)

- 日報の自動生成(M4)・週次サマリ・異常検知・ダッシュボードの集計対象の変更 — これらは
  「問いかけ」ではなくレポート/分析であり、従来どおりロール基準を維持する
- ユーザーマスタの追加・編集 UI(名前・ロール・active の変更は SQL 運用を継続)
- 本人(メンバー自身)による設定変更(管理者のみが変更できる)
- 問いかけ種別ごと(朝のみ可・状況確認のみ可等)の細分化 — 必要性が確認された時点で判断

## 7. 受け入れ基準

1. 管理者がダッシュボードから任意のユーザーの問いかけ可否を切り替えられる(SQL 不要)
2. 問いかけ不可のユーザーには朝の問いかけ・状況確認(全員/個別)のいずれも配信されない
3. 問いかけ可に設定した管理者ロールのユーザーにも問いかけが配信される
4. 既存環境への適用直後の配信対象は従来(active な member 全員)と一致する(下位互換)
5. seed テンプレートの再実行が UI で変更した可否設定を巻き戻さない
6. 閲覧専用ロール・書込ロールから checkin_enabled 以外のユーザー属性を UI 経由で変更できない

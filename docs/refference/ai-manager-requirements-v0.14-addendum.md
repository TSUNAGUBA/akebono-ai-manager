# AI マネージャー要件定義 v0.14 追補 — 随時 QA の Google カレンダー参照

> **位置づけ:** 本書は `ai-manager-requirements-design.md`(v0.2)への**追補**であり、v0.2 本文は変更しない。
> 既存の追補(v0.3〜v0.13)と本書が矛盾する場合は本書(v0.14)を優先する。
> **背景:** オペレーター報告(2026-07-13)。メンバーが Chat で「私の明日の予定を確認して」と質問したところ、
> AI が「私はあなたの明日の予定に関する情報を持っていません。個人のスケジュールは、あなたのカレンダーや
> スケジュール管理ツールでご確認ください」と回答し、low_confidence エスカレーションが起票された。
> カレンダー連携(ドメイン全体委任・`CALENDAR_ENABLED` フラグ・calendar.readonly)は実装済みだが、
> 朝の問いかけ配信(batch)にしか使われておらず、随時 QA はカレンダーを参照しない。

## 1. 背景と課題

| # | 課題 | 現状 |
|---|---|---|
| C26 | 随時 QA が本人の予定に答えられない | カレンダー取得(M2 拡張)は batch の朝の問いかけ配信専用の実装で、随時 QA(chat-gateway)への供給パスがない。委任スコープ(calendar.readonly)も配線(`CALENDAR_ENABLED`)も既存だが、メンバーの予定質問は参考情報なしの low_confidence 回答となり、答えられるはずの質問で不要なエスカレーションが起票される |

## 2. 随時 QA のカレンダー参照

### 2.1 予定質問の検知(ルールベース)

構造の判定を LLM にさせない(v0.3 設計原則2)。以下のキーワードの**いずれかを含む**メッセージを
予定質問として検知し、検知した場合のみカレンダーを参照する:

- 予定 / カレンダー / スケジュール

### 2.2 対象日の解決

| 質問中の表現 | 対象日(JST) |
|---|---|
| 明後日・あさって | 翌々日 |
| 明日・あした・あす | 翌日 |
| それ以外(今日・本日・無指定) | 当日 |

判定は**明後日 → 明日 → 既定(当日)**の優先順で行う(「明後日」を先に判定し、
「明日」系パターンとの部分一致の混同を防ぐ)。

### 2.3 供給ブロック

- 検知した場合のみ、本人(質問者)の Google カレンダー(プライマリ)から**対象日**の予定を取得し、
  system プロンプトへ「## カレンダー情報({対象日} の予定)」ブロックとして供給する
  (顧客マスタ情報 v0.7 §3 / プロジェクト情報 v0.10 §4.2 / 会話履歴 v0.12 §5 と同じ流儀)
- プロンプト指示(ADHOC_QA_INSTRUCTION)に1行追加: カレンダー情報が与えられている場合は
  本人の Google カレンダーから取得した確定情報として予定の質問に答えてよい
  (予定がない場合はその旨を伝えてよい)
- 取得は既存のカレンダー読み取り(M2 拡張・batch 実装)を shared へ共通化して使う(原則3)。
  対象日を timeMin/timeMax(JST の 0:00〜23:59)に取る形へ一般化し、
  朝の問いかけ配信は「当日を渡す薄いラッパー」で従来どおり動作する(呼び出し互換)

### 2.4 非ブロッキング

- `CALENDAR_ENABLED` が無効・取得失敗・予定質問でない場合は従来動作(ブロックなしで QA 継続)。
  カレンダーは補助文脈であり、その失敗が QA をブロックしない(開発原則4。朝の問いかけと同じ設計)

## 3. 前提

- `CALENDAR_ENABLED=true` + Workspace のドメイン全体委任(calendar.readonly)
  = deployment-setup.md **Step 7-6**(朝の問いかけのカレンダー反映と同一の設定)
- サービスアカウントは 3 サービス(chat-gateway / batch / dashboard)共通のため、
  chat-gateway 用の**追加の委任設定は不要**(デプロイ配線として `CALENDAR_ENABLED` /
  `DELEGATION_SA_EMAIL` を chat-gateway にも渡すのみ)
- 未設定環境では従来動作のまま(検知しても取得せず、ブロックなしで回答する)

## 4. スコープ外(本追補では実施しない)

- 「今週」等の期間指定・特定日付(「7月20日の予定」)の解決(当日/翌日/翌々日のみ)
- 予定の作成・変更(読み取りのみ。calendar.readonly の権限境界を維持)
- 朝夕対話・状況確認への追加供給(朝の問いかけ配信の当日予定反映は従来どおり。
  予定ゼロ件時の供給テキストの一般化のみ §5-6 に記載)
- 他人の予定の参照(委任の subject は常に質問者本人)

## 5. 受け入れ基準

1. 「私の明日の予定を確認して」に、翌日(JST)の本人カレンダーの予定で回答できる(インシデントの再現テスト)
2. 「明後日の予定は?」は翌々日の予定を参照する
3. 予定キーワードを含まない質問ではカレンダーを参照しない(不要な API 呼び出しをしない)
4. `CALENDAR_ENABLED` 無効時はブロックなしで従来どおり回答する
5. カレンダー取得に失敗しても QA は継続する(非ブロッキング)
6. 下位互換: スキーマ変更なし・既存データへの影響なし(朝の問いかけ配信のカレンダー反映は
   従来どおり動作する。データ更新パッチ不要)。ただし対象日の一般化に伴い、予定ゼロ件時の
   供給テキストのみ「(本日の予定はありません)」→「(この日の予定はありません)」に変更した
   (LLM への文脈テキストであり利用者向け出力ではないため、文面生成への実質影響なしと判断 —
   原則7 に基づく変更内容の明示。パッチ不要)

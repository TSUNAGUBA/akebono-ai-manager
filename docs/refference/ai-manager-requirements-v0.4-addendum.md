# AI マネージャー要件定義 v0.4 追補 — ナレッジ管理 UI

> **位置づけ:** v0.2 本文・v0.3 追補への追加要件。矛盾時は本書を優先する。
> **背景:** オペレーター要望(2026-07-10)「ナレッジのドキュメント格納も画面から行いたい」。

## 1. 要件

ダッシュボードの管理者限定ページに **ナレッジ管理**(`/admin/knowledge`)を追加する。

| 機能 | 内容 |
|---|---|
| 一覧 | ナレッジフォルダ(Drive)配下の文書一覧(パス・名前・更新日時)+同期状態(rag のチャンク数・最終同期日時) |
| 投入 | 格納先(共通 judgement / 業界 domain/{業界} / 顧客 customer/{顧客ID})+ファイル名+本文(テキスト)を指定して Drive へ保存。業界・顧客はマスタから選択 |
| 上書き | 同一フォルダ・同名ファイルが存在する場合は内容を更新(新規作成しない) |
| 削除 | Drive のゴミ箱へ移動(復元可能な安全側の削除)。チャンクは次回同期で掃除される |
| 即時同期 | 「今すぐ同期」ボタンで knowledge-sync ジョブを起動(毎日 06:30 を待たずに反映) |

## 2. 設計原則(確定判断)

1. **SoT は引き続き Drive。** UI は Drive への投入経路にすぎず、保存 → knowledge-sync → rag の
   既存データフローを変えない(原則 6)。UI からの直接 rag 書込は行わない(SoT 分裂の防止)。
   Drive 上での直接編集も従来どおり可能(UI と Drive のどちらで編集しても同期が吸収する)
2. **書込はランタイム SA 自身の権限で行う**(ドメイン全体委任は使わない)。SA がアクセスできるのは
   明示的に共有されたフォルダのみであり、共有 ACL が実効的な権限境界になる
3. 削除は物理削除ではなくゴミ箱移動(オペレーターの誤操作から復元可能)
4. 既存の管理 UI 規約(admin 限定・CSRF・監査ログ・入力検証・レスポンシブ)をすべて踏襲する

## 3. 前提の変更(オペレーター作業)

- ナレッジフォルダの共有を、ランタイム SA に対して**閲覧者 → 編集者**へ変更する(Step 7-3 改訂)
- 未変更の間、一覧は動作し、投入・削除は Drive API の権限エラーを画面に表示する(閲覧機能は止めない)

## 4. スコープ外

- Google ドキュメント形式での新規作成(投入はテキスト/Markdown ファイルとして保存する。
  既存の Google ドキュメントの閲覧・同期は従来どおり対象)
- Chat からのナレッジ投入(「これナレッジに入れて」)— 引き続き将来対応
- 文書のリッチエディタ(本文はプレーンテキスト入力)

## 5. 受け入れ基準

1. 管理者が画面から業界・顧客を選んで文書を投入し、「今すぐ同期」後に Chat の QA で参照される
2. 同名ファイルへの再投入が上書きになる(重複ファイルが生えない)
3. フォルダ共有が閲覧者のままの場合、投入時に権限不足である旨と対処(Step 7-3)が画面に表示される
4. 削除した文書が Drive のゴミ箱にあり、同期後に検索へ出なくなる
